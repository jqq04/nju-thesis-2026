# 项目手册

# 项目摘要

随着互联网广告与商业化业务的快速发展，广告产品形态、业务规则及运营策略日益复杂，围绕广告投放、产品说明、业务规范与运营实践等内容不断沉淀，形成了规模庞大、类型多样的专业内容资产。这类内容在业务人员培训、运营决策支持以及客户服务等场景中发挥着重要作用。然而，随着业务线数量和内容规模的持续增长，现有内容平台在内容统一管理、高效应用以及质量保障等方面逐渐暴露出诸如内容分散、复用困难、维护成本高以及质量难以评估和治理等问题，已难以满足广告与商业化场景下的实际需求。

针对上述问题，本文在分析现有内容管理与知识平台相关工作的基础上，结合企业内部广告与商业化业务的实际应用场景，设计并实现了一套面向广告与商业化内容的管理、应用与治理平台。平台以内容全生命周期为核心，对内容的建模、组织、应用与治理进行系统化设计，旨在提升内容资产的统一管理能力、应用效果以及质量可控性。

在内容管理方面，平台提出了一种基于空间与应用分层的内容组织模型，实现对不同广告业务线及其应用场景的隔离与统一管理，并支持文章、问答、课程和案例等多种内容形态的规范化管理。结合类目树、标签体系、权限控制与审批流程，平台有效降低了内容维护成本，提升了内容资产的复用性与一致性。在内容应用方面，平台在支持传统页面化内容展示的基础上，引入基于检索增强生成（Retrieval-Augmented Generation，RAG）的智能问答能力，通过内容切片、向量索引、多通道召回与重排等技术手段，为上游智能应用提供高相关性的内容支持，拓展了内容资产的应用方式。在内容治理方面，针对内容规模扩大带来的质量问题，本文构建了一种基于重复度、歧义度与覆盖度的内容治理机制，结合向量检索与大语言模型实现对内容质量的自动检测，并通过线上化治理任务实现内容治理的闭环处理。此外，平台通过数据洞察模块对内容使用效果和治理结果进行统计分析，为内容优化与治理策略调整提供数据支持。

实际应用结果表明，本文所设计和实现的平台在广告与商业化业务场景中有效提升了内容管理效率与应用效果，降低了内容治理的人力成本，对面向广告与商业化内容的内容平台建设具有一定的工程实践价值和参考意义。

关键词：广告与商业化内容；内容管理平台；检索增强生成（RAG）；内容治理；智能问答

# 功能与技术细节

## 用户角色

- 平台负责人 & 平台的最高管理角色，负责整个平台的基础配置、空间管理及全局权限控制，保障平台稳定运行。 
- 业务负责人 & 负责某一业务线或业务空间下内容体系的管理与运维，对内容结构、应用配置及业务权限进行整体把控。 
- 业务运营 & 负责平台内容的创建、编辑和维护等日常运营工作，并在权限范围内查看相关内容使用数据。 
- 客户 & 平台内容的消费角色，包括广告主、代理商等，通过页面浏览、搜索或智能助手等方式使用平台内容。

## 内容管理模块

内容管理模块面向广告与商业化场景下多业务线、多内容形态的复杂管理需求，提供统一、可扩展的内容管理能力，支持内容从创建、维护到上线的全流程管理。

### 空间与应用管理

  平台通过“空间（Space）”与“应用（Application）”两级结构对内容进行组织。其中，空间用于区分不同广告与商业化业务线，应用用于表示业务线下的具体业务领域或使用场景，从而实现内容在不同业务维度下的隔离与管理。

### 统一内容模型【核心】

  平台支持多种内容形态的统一管理，包括文章、问答、课程和案例，提供内容的创建、编辑、删除、上下线以及批量操作等功能。

  每个内容类型都有其特定的属性和操作方式，平台通过统一的内容模型对这些内容进行抽象和表示，确保不同内容类型之间的一致性和可扩展性。

  - 文章（knowledge）： 以文档形式存在，支持文字、图片、表格等多种富文本内容。一般为业务中沉淀的广告产品说明、业务规范、运营实践等内容。
  - 问答（faq）：由标准问、相似问与答案构成。一般是运营同学根据历史问题和客户反馈，手动编写的较为标准的问答内容。
  - 课程（course）： 除课程基本信息外，由多个课节组成，每个课节支持视频或文档形式的内容。课程一般为广告产品的培训课程、业务规范的讲解课程等。
  - 案例（case）： 以非结构化数据形式存在，支持自定义字段配置，以满足不同业务案例的表达需求。一般以真实场景或案例形式存在，展示业务实践中的成功案例。

  数据库层上，由一张内容表（Content）维护所有内容的元数据，包括内容ID、类型、标题、摘要、内容体、创建时间、更新时间、状态等字段。不同内容类型的具体属性则存储在对应的子表中，如文章（Knowledge）、问答（Faq）、课程（Course）、案例（Case）等。其中案例较为特殊，对应子表在MongoDB中（以适应结构化数据）。

### 权限管理
  平台的权限有两种类型，第一种为平台管理权限，第二种为内容权限。
#### 平台管理权限
  包括对平台全局配置、空间管理、应用管理、用户管理等方面的权限。基于角色的访问控制（RBAC）体系，在空间与应用维度实现权限隔离与精细化授权。权限粒度应覆盖模块级与操作级，既能控制用户进入某一空间/应用的访问权限，也能控制内容的创建、编辑、删除、提审、审批、上线与下线等具体操作权限。权限管理功能主要包含成员角色分配与权限策略配置等

  平台角色包括平台负责人、业务负责人、业务运营三种。

平台管理权限的实现以“角色-权限项-作用域”三元组为核心：权限项表达“能做什么”，角色表达“是谁”，作用域表达“在什么范围内生效（全局/空间/应用）”。平台侧的RBAC仅用于平台管理与运营后台的权限控制；内容级的可见性与细粒度访问控制由content\_auth负责，两者在读写链路中叠加生效。

**1）权限项与权限码设计**

平台将管理能力拆分为模块级与操作级两层权限项，并为每个权限项分配稳定的权限码（permission\_code），用于后端鉴权与前端菜单/按钮控制的一致口径。典型权限码如下：

- 平台配置：PLATFORM\_CONFIG\_READ / PLATFORM\_CONFIG\_WRITE  
- 空间管理：SPACE\_CREATE / SPACE\_UPDATE / SPACE\_MEMBER\_MANAGE  
- 应用管理：APP\_CREATE / APP\_UPDATE / APP\_MEMBER\_MANAGE  
- 内容管理（管理侧）：CONTENT\_CREATE / CONTENT\_EDIT / CONTENT\_DELETE / CONTENT\_SUBMIT\_REVIEW / CONTENT\_APPROVE / CONTENT\_PUBLISH / CONTENT\_OFFLINE  
- 权限管理：ROLE\_MANAGE / ROLE\_ASSIGN / PERMISSION\_QUERY  
- 用户管理：USER\_QUERY / USER\_DISABLE  

其中“内容管理（管理侧）”权限码用于控制运营后台是否允许执行内容写操作；是否允许访问某条内容、以及是否允许在问答/检索中返回某条内容，仍需要通过content\_auth进行对象级判断。

**2）数据模型与表结构**

RBAC实现落地为以下核心数据对象（可采用MySQL存储）：

- user：用户基础信息（user\_id、name、status等）。
- role：角色定义（role\_id、role\_name、role\_type），role\_type区分平台内置角色与空间/应用自定义角色。
- permission：权限项定义（permission\_id、permission\_code、module、action）。
- role\_permission：角色与权限项的多对多关联。
- user\_role\_binding：用户与角色绑定关系，包含作用域字段（scope\_type、scope\_id）。
  - scope\_type：GLOBAL / SPACE / APP
  - scope\_id：当scope\_type为SPACE/APP时分别对应space\_id或app\_id；GLOBAL可为空或固定为0

通过scope字段，平台能够表达同一用户在不同空间/应用下拥有不同管理权限的现实需求，达到隔离与最小授权原则。

**3）平台内置角色的权限边界**

- 平台负责人：绑定GLOBAL作用域，拥有平台全局配置、空间创建与全局权限管理等能力；可进入所有空间与应用进行应急处置。
- 业务负责人：通常绑定SPACE作用域，拥有该空间下应用管理、成员管理、内容审批与发布等能力；可管理空间内的权限策略与角色分配。
- 业务运营：通常绑定APP或SPACE作用域，拥有内容创建编辑、提审、组织维护（类目/标签）等能力；不具备全局配置与跨空间管理权限。

**4）鉴权流程（管理侧）**

平台管理接口在进入业务逻辑前统一进行鉴权，流程如下：

1. 身份解析：从请求上下文解析user\_id与组织信息（如部门、员工号等）。
2. 作用域确定：根据请求参数识别scope\_type与scope\_id（如space\_id、app\_id），并校验用户是否具备进入该范围的最基本访问资格。
3. 权限校验：调用checkPermission(user\_id, scope\_type, scope\_id, permission\_code)，通过user\_role\_binding与role\_permission计算是否命中授权。
4. 审计记录：对空间/应用配置变更、成员变更、角色分配、审批发布等关键操作记录审计日志（操作人、对象、时间、变更前后摘要）。

为降低鉴权开销，可对user\_id在不同scope下的权限集合进行缓存，并在角色绑定或权限策略变更后进行缓存失效。

**5）成员与角色分配机制**

平台提供成员管理与角色分配能力，支撑“空间/应用”的组织边界落地：

- 创建空间：初始化默认角色与默认权限策略，并将创建人绑定为该空间的业务负责人（SPACE作用域）。
- 创建应用：在空间内初始化应用配置，并将创建人绑定为该应用的业务运营或应用管理员（APP作用域），同时继承空间层的最小可用权限集合。
- 成员变更：业务负责人可在空间/应用维度为成员分配角色；平台负责人可跨空间处理成员与权限异常问题。
  
#### 内容权限
  包括所有内容的创建、编辑、删除、上下线等操作权限，以及对客端的内容展示权限。采用content_auth模块实现，基于用户角色和内容属性进行权限判断。
  content_auth表结构如下：
| 字段名        | 描述           | 类型        | 备注                                      |
|---------------|----------------|-------------|-------------------------------------------|
| id            | 主键 ID        | bigint      | 自增主键                                  |
| content_id    | 内容 ID        | bigint      | 被授权对象的唯一标识                      |
| auth_tactic   | 权限策略       | varchar(50) | 策略类型（如员工匹配、部门匹配等）        |
| auth_variable | 权限变量       | varchar(200)| 策略变量值（如员工 ID、部门 ID 等）       |
| auth_code     | 操作权限       | varchar(20) | 权限码集合（如 r/w/u 等）                 |
