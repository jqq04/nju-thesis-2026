## 项目摘要

随着互联网广告与商业化业务的快速发展，广告产品形态、业务规则及运营策略日益复杂，围绕广告投放、产品说明、业务规范与运营实践等内容不断沉淀，形成了规模庞大、类型多样的专业内容资产。这类内容在业务人员培训、运营决策支持以及客户服务等场景中发挥着重要作用。然而，随着业务线数量和内容规模的持续增长，现有内容平台在内容统一管理、高效应用以及质量保障等方面逐渐暴露出诸如内容分散、复用困难、维护成本高以及质量难以评估和治理等问题，已难以满足广告与商业化场景下的实际需求。

针对上述问题，本文在分析现有内容管理与知识平台相关工作的基础上，结合企业内部广告与商业化业务的实际应用场景，设计并实现了一套面向广告与商业化内容的管理、应用与治理平台。平台以内容全生命周期为核心，对内容的建模、组织、应用与治理进行系统化设计，旨在提升内容资产的统一管理能力、应用效果以及质量可控性。

在内容管理方面，平台提出了一种基于空间与应用分层的内容组织模型，实现对不同广告业务线及其应用场景的隔离与统一管理，并支持文章、问答、课程和案例等多种内容形态的规范化管理。结合类目树、标签体系、权限控制与审批流程，平台有效降低了内容维护成本，提升了内容资产的复用性与一致性。在内容应用方面，平台在支持传统页面化内容展示的基础上，引入基于检索增强生成（Retrieval-Augmented Generation，RAG）的智能问答能力，通过内容切片、向量索引、多通道召回与重排等技术手段，为上游智能应用提供高相关性的内容支持，拓展了内容资产的应用方式。在内容治理方面，针对内容规模扩大带来的质量问题，本文构建了一种基于重复度、歧义度与覆盖度的内容治理机制，结合向量检索与大语言模型实现对内容质量的自动检测，并通过线上化治理任务实现内容治理的闭环处理。此外，平台通过数据洞察模块对内容使用效果和治理结果进行统计分析，为内容优化与治理策略调整提供数据支持。

实际应用结果表明，本文所设计和实现的平台在广告与商业化业务场景中有效提升了内容管理效率与应用效果，降低了内容治理的人力成本，对面向广告与商业化内容的内容平台建设具有一定的工程实践价值和参考意义。

关键词：广告与商业化内容；内容管理平台；检索增强生成（RAG）；内容治理；智能问答

## 功能与技术细节

### 用户角色

- 平台负责人：平台最高管理角色，负责平台基础配置、空间管理与全局权限控制，保障平台稳定运行。
- 业务负责人：负责某一业务线或业务空间下内容体系的管理与运维，对内容结构、应用配置及业务权限进行整体把控。
- 业务运营：负责内容创建、编辑和维护等日常运营工作，并在权限范围内查看相关内容使用数据。
- 客户：内容消费角色，包括广告主、代理商等，通过页面浏览、搜索或智能助手等方式使用平台内容。

### 内容管理模块

内容管理模块面向广告与商业化场景下多业务线、多内容形态的复杂管理需求，提供统一、可扩展的内容管理能力，支持内容从创建、维护到上线的全流程管理。

#### 空间与应用管理

平台通过“空间（Space）”与“应用（Application）”两级结构对内容进行组织。其中，空间用于区分不同广告与商业化业务线；应用用于表示业务线下的具体业务领域或使用场景，从而实现内容在不同业务维度下的隔离与管理。

#### 统一内容模型【重要】

平台支持多种内容形态的统一管理，包括文章、问答、课程和案例，提供内容的创建、编辑、删除、上下线以及批量操作等功能。

平台通过统一内容模型对不同内容类型进行抽象与表示，确保不同内容类型之间的一致性与可扩展性。

- 文章（knowledge）：以文档形式存在，支持文字、图片、表格等多种富文本内容。一般为业务中沉淀的广告产品说明、业务规范、运营实践等内容。
- 问答（faq）：由标准问、相似问与答案构成。一般为运营同学根据历史问题和客户反馈，手动编写的较为标准的问答内容。
- 课程（course）：除课程基本信息外，由多个课节组成，每个课节支持视频或文档形式的内容。课程一般为广告产品的培训课程、业务规范的讲解课程等。
- 案例（case）：以非结构化数据形式存在，支持自定义字段配置，以满足不同业务案例的表达需求。一般以真实场景或案例形式存在，展示业务实践中的成功案例。

数据库层上，由一张内容表（Content）维护所有内容的元数据，包括内容 ID、类型、标题、摘要、内容体、创建时间、更新时间、状态等字段。不同内容类型的具体属性则存储在对应的子表中，如文章（Knowledge）、问答（Faq）、课程（Course）、案例（Case）等。其中案例较为特殊，对应子表在 MongoDB 中（以适应结构化数据）。

内置三种角色：

- 平台负责人（GLOBAL）：全局配置、空间创建、应急处置。
- 业务负责人（SPACE）：空间内应用/成员/审批/发布。
- 业务运营（APP|SPACE）：内容增删改、提审、类目标签维护。

权限码按“模块_操作”命名（如 `SPACE_CREATE`、`CONTENT_PUBLISH`），统一拦截器按 `user_id + scope + code` 做缓存鉴权。

创建空间/应用时自动初始化角色并绑定创建人；后续成员变更由空间负责人自助分配角色，平台负责人可跨空间处理异常。

内容级可见性由独立 `content_auth` 模块再判一层，与平台权限叠加生效。

##### 内容权限（content_auth）

内容权限涵盖内容全生命周期的操作与展示控制，包括创建、编辑、删除、上下线等操作权限，以及面向客户端的可见性与可操作范围。系统通过 `content_auth` 模块统一实现权限判断：在访问者发起内容操作或请求内容展示时，根据访问者角色/身份属性与内容绑定的授权规则进行匹配，从而决定是否允许执行对应动作。

`content_auth` 表结构如下：

| 字段名          | 描述     | 类型         | 备注                                           |
|----------------|----------|--------------|------------------------------------------------|
| id             | 主键 ID  | bigint       | 自增主键                                       |
| content_id     | 内容 ID  | bigint       | 被授权对象的唯一标识                           |
| auth_tactic    | 权限策略 | varchar(50)  | 策略类型（如员工匹配、部门匹配等）             |
| auth_variable  | 权限变量 | varchar(200) | 策略变量值（如员工 ID、部门 ID 等）            |
| auth_code      | 操作权限 | varchar(20)  | 权限码集合（如 `r/w/u` 等）                    |

`auth_tactic` 权限策略说明：

1. 免登录可见：无需登录即可访问。
2. 所有外部用户可见：任意外部用户均可访问。
3. 所有内部员工可见：任意内部员工均可访问。
4. 员工匹配（employee）：当访问者员工 ID 与 `auth_variable` 完全一致时命中。
5. 部门匹配（department）：当访问者所属部门 ID（或其上级链路中的任一祖先部门 ID）与 `auth_variable` 一致时命中。
6. 角色匹配（role）：当访问者角色 ID 与 `auth_variable` 一致时命中。
7. 自定义匹配（custom）：按自定义表达式判断是否命中；表达式可引用访问者属性（如部门、岗位等）。

在业务层面，可以把 `content_auth` 理解为“内容与访问规则的绑定清单”。每条内容在发布或配置权限时，会生成若干条授权规则（不同的 `auth_tactic + auth_variable` 组合），并声明这些规则对应允许的操作集合（`auth_code`）。当用户尝试查看、编辑、删除或上下线等动作时，系统会以“内容 + 动作 + 访问者属性”为输入，遍历并匹配这些规则：只要命中任一允许该动作的规则即可放行；若没有任何规则命中，则拒绝访问或隐藏对应能力。这样既能支持全员可见等通用策略，也能按员工、部门、角色或自定义条件实现精细化控制。

#### 内容上线流程【重要】

1. 校验内容是否完整。
2. 校验发布者是否拥有对应的权限。
3. 发起内容发布审批流程，数据库记录审批状态与审批单。
4. 审批服务调用消息中心，通知审批人进行审批。
5. 审批人在审批平台进行审批操作，记录审批结果与审批人。
6. 审核通过/拒绝：更新审批状态，通知发布者审批结果。
7. 审批服务回调内容服务。
8. 内容服务更新内容状态为已发布，更新相关信息，同时发布更新事件到 MQ。

### 内容应用模块

#### 统一内容索引与过滤（ES核心定位）

本平台采用 **Elasticsearch (ES)** 作为统一的内容索引与全量过滤中心，而非仅用于全文检索。其核心定位如下：

1.  **全量信息同步（Write）**：
    *   后台内容的创建、修改、更新、发布、下线、权限变更等所有操作，均通过发送 `content_update` MQ 消息触发同步。
    *   消费端收到消息后，回查 DB 获取最新的**全量信息**（含内容主体 + 扩展元信息 + 权限规则 `content_auth` + 标签/类目/空间ID等），将其扁平化写入 ES 文档。
    *   ES 充当了“宽表”角色，承载了所有维度的筛选能力。

2.  **统一过滤与圈选（Read - List/Filter）**：
    *   **页面展示（列表/类目/详情/管理台）**：所有涉及内容筛选、排序、权限过滤的接口，**均优先查询 ES**。利用 ES 强大的倒排索引，一次性完成 `status`、`category`、`tag` 以及 `auth_match`（权限匹配）等复杂条件的圈选。
    *   **详情回查**：ES 返回满足条件的 `content_id` 列表（及分页信息）后，服务端再根据 ID 批量回查 DB（或 Redis）获取实时、强一致的内容详情（如正文大字段、最新状态），组装返回给前端。

3.  **全文检索与高亮（Read - Search）**：
    *   在搜索场景下，继续发挥 ES 的全文检索（Match/QueryString）与高亮（Highlight）强项，同时复用上述的统一过滤逻辑。

这种架构确保了：
*   **过滤性能**：利用 ES 处理复杂的组合查询与权限筛选，避免 DB 慢查询。
*   **数据一致性**：详情展示依然走 DB，保证用户看到的具体内容（如正文、标题）是绝对实时的，避免 ES 同步延迟带来的详情页数据回滚或旧数据问题。

#### 页面化展示

基于上述统一索引架构，支持在搭建的页面中以多种形式展示内容：

- 内容列表展示（走 ES 过滤 + DB 详情）
- 基于目录（类目树）的内容展示（走 ES 过滤 + DB 详情）
- 基于标签的内容分类展示（走 ES 过滤 + DB 详情）
- 内容详情展示（走 ES/DB 权限校验 + DB 详情）
- 内容搜索（走 ES 检索/高亮 + DB 详情补全）



#### 为智能问答提供的RAG能力

##### 内容切片与向量索引【重要】

内容切片指在前置处理阶段对不同内容形态进行切片处理，使内容能够被有效切分为可检索的语义单元（切片），每个切片包含足够上下文信息，能够独立被理解和处理。

1. 文章：以固定长度为基础进行切片，同时尽量保持段落语义完整性，避免跨段切分导致上下文语义断裂；切片携带文章标题、摘要、标签等元数据。
2. 问答：问题与答案作为完整语义单元不进行切片处理，分别入库并建立索引；召回阶段根据不同应用场景进行组合使用（如只对问题检索，或同时对问题与答案检索）。
3. 课程：考虑到成本，通过 LLM 提取课节字幕内容，一个课节作为一个切片；切片携带课程与课节的标题、摘要、标签等元数据。
4. 案例：每个案例作为一个切片；切片携带案例标题、摘要、标签等元数据。

另外两种特殊类型在切片处理时需要考虑其特殊性：

5. 图片：
   - 使用多模态 Embedding 模型，使图片与文本内容在向量空间等价；
   - 通过 LLM 对图片进行语义理解，将生成的文本化理解结果与图片链接及相关上下文信息一并入库，提升图片类信息在检索与问答场景下的可用性。
6. 表格：避免采用默认的纯文本切分方式，保证按行语义完整切分，并在每个切片中携带对应表头信息，以保持表格语义结构，提升表格类知识在智能问答中的召回质量和理解准确性。

同时，每个切片在索引时也需要携带其所属的内容 ID、应用 ID、内容权限等元数据，以便在召回阶段进行过滤与权限校验。

索引时，会分别进行ES和Embedding索引，以支持基于文本的召回与基于语义的召回。前者侧重字面匹配，适合精准关键词场景；后者侧重语义相似，可捕捉同义表达与隐含意图。两条通道互补，既保证高准确，也兼顾高召回，为后续问答体验提供多一层保障。

##### 内容召回【重要】

内容召回指在用户查询或发起智能问答时，根据查询/问题的语义，从已索引的内容切片中提取与查询相关的内容片段。召回流程如下：

1. 问题改写：将用户问题改写为更标准化的表达，以提高召回质量。
2. 多通道召回：从多个索引通道中提取切片（如文章索引、问答索引、课程索引等），组合形成候选结果；同时支持语义（Embedding）与文本（ES）两种召回方式。
3. 重排与过滤：在多通道召回基础上对候选结果重排，并在必要时进行过滤，以确保最相关的切片排在前面。

在召回过程中，比较重要的参数有：

- `TopK`：返回 `TopK` 个最相关的切片。
- `MinScore`：最小相似度阈值，只有相似度大于等于该阈值的切片才会被返回。

### 内容治理模块

#### 内容重复度/歧义度模型【重要】

随着企业业务快速发展和内容规模扩大，内容质量问题日益凸显，其中内容重复与歧义尤为突出，严重影响内容中台的服务质量与用户体验。平台通过重复度与歧义度检测识别问题内容，并支撑后续治理闭环。

重复度治理用于检测不同内容之间的文本或语义相似度，识别内容重复问题，预防信息冗余。内容重复判断标准如下：

- 完全重复：两篇文章几乎完全一致，包括用词、顺序等。
- 包含重复：两篇文章主体相同，且一篇文章的描述部分基本都能在另一篇文章中找到类似部分。
- 极度重复：两篇文章主体相同，描述属性重复部分占比在 80%–100%，仅有少量差异。
- 中度重复：两篇文章主体相同，描述属性重复部分占比在 50%–80%，其余部分不同。
- 轻微重复：两篇文章主体相同，描述属性重复部分占比在 25%–50%。
- 不重复：两篇文章主体不相同，或描述属性重复部分不到 25%。

歧义度治理用于检测描述同一业务问题的不同内容之间是否存在语义冲突或不一致，降低内容歧义对用户理解的影响。

实现方案：

1. 遍历所有内容。
2. 对每个内容使用 RAG 召回其相关内容 `TopK`。
3. 使用 LLM + `prompt` 判断相似度/歧义度并打分。
4. 对所有内容进行重复度/歧义度统计分析，得到重复度/歧义度分布情况。

#### 内容覆盖度检测【重要】

覆盖度治理用于分析现有内容对广告与商业化知识体系的覆盖情况，辅助发现内容缺失或薄弱领域。

实现方案：

1. 接入广告与商业化知识体系知识图谱，包括类目/标签/概念/定义等。
2. 遍历知识体系中的类目/标签/概念/定义等。
3. 对每个类目/标签/概念/定义，使用 RAG 召回其相关内容 `TopK`。
4. 使用 LLM + `prompt` 判断内容是否覆盖该类目/标签/概念/定义并打分。
5. 对所有类目/标签/概念/定义进行覆盖度统计分析，得到覆盖度分布情况。

辅助方案：

1. 汇总用户 `query`，过滤噪声或无关 `query`，仅保留与广告与商业化知识体系相关的 `query`。
2. 对每个 `query` 使用 RAG 召回其相关内容 `TopK`。
3. 使用 LLM + `prompt` 判断内容知识是否能满足用户 `query` 需求并打分。
4. 对所有 `query` 进行相关度统计分析，得到相关度分布情况。

#### 治理任务线上化【重要】

平台基于上述三种模型以及用户反馈自动检测，生成线上治理任务，并指派至对应业务人员，实现内容治理的闭环处理。

1. 主任务：治理任务的上层管理单元，负责任务创建、配置和状态管理。主任务创建流程：用户在空间层新增“任务”菜单，选择“新建任务”，填写任务名称、选择业务空间、设置会话周期和召回阈值；系统执行内容补充任务计算，主任务状态变为“计算中”；计算完成后主任务状态变为“计算完成”，并生成治理任务列表。
2. 重复歧义任务：用于识别与处理内容重复、歧义问题，内容重复与歧义判定规则同现在线下跑数规则。对比本应用下全部上线状态的文章/问答与内容中台全量上线状态内容，生成重复内容对与歧义内容对。
3. 覆盖度任务：用于分析现有内容对广告与商业化知识体系的覆盖情况，辅助发现内容缺失或薄弱领域。

重复歧义任务处理机制：

- 人工标记处理结果：人工判断是否需要内容下线并完成任务标记；若存在需要下线的内容，可下线本应用下重复内容。
- 系统自动巡检：系统每天定时巡检，检查自上次巡检之后发生内容变更/下线/删除的内容是否存在关联待处理任务；LLM 判断是否仍存在重复、歧义问题，若不存在则系统自动将任务标记为已完成状态。

覆盖度任务处理机制：

- 人工标记处理结果：对缺失知识人工判断是否需要内容补充并完成任务标记；若需补充，可在平台新增相关内容。
- 系统自动巡检：系统每天定时巡检，检查自上次巡检之后发生内容变更/上线的内容；LLM 判断是否补充了缺失知识，若已补充则系统自动将任务标记为已完成状态。

# 背景
你是一个专业的语言学家，你的任务是根据提炼规则，提炼出两组内容的**主体**，并根据步骤分析两组内容是否存在相同的主体，如果存在则进一步判断两组内容对同一**主体**的**主体描述**（围绕特定主体的具体陈述或名词解释）是否一致。

# 概念定义
- **主体**可以是平台名称、功能名称、业务指标、用户问题等等，每组内容都可以有多个**主体**。
- **主体描述**是指围绕**主体**的具体陈述或名词解释。
- **业务场景**是指主体所在的业务领域，通常为平台名称，界定了**主体**的影响范围。

# 分析步骤
请按照以下步骤进行分析：
1. 仔细阅读两组内容，识别出两组内容是否具有相同的**主体**，以下是识别**主体**的具体规则：
  - 识别**主体**时，要结合上下文所提到的业务领域、行业，例如“经营线索”和“抖音来客的经营线索”，前者主体为“经营线索”，而后者限定了“抖音来客”这个业务领域，因此**主体**为“抖音来客经营线索”。
  - 识别**主体**时，要结合上下文所提到的活动，例如“春上新商品红包”和“38好物节商品红包”，虽都为红包，但两者因为所处活动不同，需识别为不同的**主体**。
  - 识别**主体**时，要结合上下文所提到的行业，例如“美妆红包”和“医疗红包”，虽都为红包，但行业不同，需识别为不同的**主体**。
  - 识别**主体**时，要结合上下文所提到的限定词和来源，例如“平台优惠券”和“智能优惠券”，虽都为优惠券，但限定词不同，需识别为不同的**主体**。
  - 识别**主体**时，要结合上下文所提到的功能，例如“待开票账单”和“待开票结算单”，都是待开票相关业务，但“账单”和“结算单”是不同的单据类型，需识别为不同的**主体**。
  - 识别**主体**时，要避免将**主体**所涉及的规则、机制等单独识别为**主体**，应全面结合业务场景和整体功能进行准确判断。
  - 识别**主体**和**主体描述**时，仅从内容中提炼直接、明确、无歧义地陈述的事实性信息，避免任何形式的推断、解读或基于上下文进行的猜测。
2. 以下是判断两组内容**主体**是否相同的具体规则，必须严格按照规则进行判断：
  - 相同的**主体**，其功能、场景、目标、行业必须一致，不能自行推断或模糊联想为一致。
  - 相同的**主体**，需要有类似的**业务场景**，部分业务场景可能使用缩写，例如“千川-客服Copilot-对客”和“巨量千川帮助中心”，都属于“千川”业务场景。
  - 对于表述类似但本质不同的**主体**要准确区分，例如“视频”和“图文视频”不是相同的**主体**，例如“直播全域推广”和“直播间”不是相同的**主体**。
3. 请着重确保**主体**判断的准确性，因为**主体**判断失误将会直接影响最终结果。
4. 两组内容必须有相同的**主体**才需要继续下一步的分析，如果识别出没有相同的**主体**，则跳过后续步骤，直接输出结果，判定为一致。
5. 当确定两段内容**主体**相同时，需进一步判断对于相同**主体**的**主体描述**是否一致，以下是判断两组**主体描述**是否一致的具体规则，必须严格按照规则进行判断：
  - **判断的前提条件**：必须以直接、明确、无歧义地陈述的事实性信息进行判断，避免任何形式的推断、解读或基于上下文的猜测。
  - **主体描述**陈述的内容存在明确的、直接的对立关系时，判定为不一致（例如：“支持”与“不支持”），若只是补充关系或者因为两段描述不在同一逻辑层面上时（例如一组是总体性的说明，另一组是具体例外性的说明），不应判定为不一致。
  - **主体描述**判断时要基于文本明确表述，对于另一方未提及的情况，不能主观认定对立；对于未明确具体指代的表述，不能轻易与其他明确表述判定为对立，需确保对比的**主体描述**在**主体**指代上清晰一致。例如一组内容明确某主体的某规则，另一组未提及该规则，不能判定两者描述存在对立，应视为一致。
  - **主体描述**中功能的上线时间、放量时间、披露时间或覆盖时间不同时，判定为不一致（例如“2024年8月1日之后”与“2024年8月20日开始”，以及“11月18日逐步覆盖”与“11月底全部覆盖”）。
  - **主体描述**对相同**主体**的数值统计口径、数值维度、公式组成不同时，判定为不一致（例如“>80”与>4.8”，“A=B+C”与“A=B+C+D”）。
  - **主体描述**的功能数量、行业数量不一致时，判定为不一致（例如“覆盖了家具等5大行业”与“新增通信行业至6大行业”），但如果核心功能相同，仅列举数量方面存在差异，应视为一致。
  - **主体描述**的活动时间点或时间范围不一致，且不存在包含关系时，判定为不一致（例如“第2~3个自然日”与“第4个自然日”）。
  - **主体描述**中若一方为普遍适用描述，另一方为普遍适用基础上的部分限制适用描述，若无明确冲突，不应判定为不一致。例如“适合所有商家使用”与“部分特殊类型商家暂不支持使用”，若整体功能实质相同，不应判定为不一致。
  - **主体描述**中存在类似描述时，要综合考虑整体语义和业务逻辑来判断是否冲突。若从整体业务逻辑来看不存在矛盾，则不应判定为不一致。
  - **主体描述**中涉及到“原xx”的描述，说明功能名称变更或已下线，可忽略该部分判断。
  - **主体描述**中有关举例、示例、特殊场景、“例如”、“如”或URL链接这类内容，可忽略该部分判断。
  - **主体描述**中以下不同用词但词语本身含义相同，可以视为一致：
    - 视为相同含义的用词：“使用”、“上线”、“上传”、“开启”、“启用”、“开放”。
    - 视为相同含义的用词：“下线”、“暂停”。
6. 根据上述所有规则，结合业务主题、业务场景和功能等关键信息重新检查识别结果，准确判断**主体**是否相同以及描述是否存在不一致。

### 数据洞察模块

本模块面向业务/运营提供“可观测 + 可分析 + 可追溯”的闭环能力。整体实现上采用“前端埋点与服务端事件采集 -> 数仓统一加工与指标口径沉淀 -> 洞察服务查询与权限控制 -> 前端可视化与下钻明细”的链路。数据加工与口径沉淀由数仓侧负责，本项目侧重点在事件规范、采集链路、与数仓/BI 的对接与查询服务建设。

核心组件与对接方式：

- 事件规范：定义统一事件模型（如 `page_view`、`click`、`query_submit`、`rag_recall`、`llm_generate`、`handoff_intent`、`handoff_success` 等），约定公共维度（`app_id`、`space_id`、`user_id`/匿名 `visitor_id`、`session_id`、`trace_id`、`content_id`、`timestamp`、`channel`、`device` 等），并维护版本与兼容策略。
- 采集链路：前端 SDK 埋点 + 服务端关键节点埋点（RAG 召回、生成、转人工等），通过日志/消息队列进入数仓侧 ODS；服务端埋点负责保证关键业务事件不丢失，并与 `trace_id` 关联，支持端到端追踪与明细还原。
- 指标查询：洞察服务提供只读 API（按天/小时聚合、按空间/渠道/页面过滤、下钻明细），对接数仓侧提供的宽表或指标层；热点查询使用缓存/物化视图减少延迟，查询侧做口径校验与限流。

技术实现思路：

- 埋点采集：在“页面曝光、组件曝光、按钮点击”等关键交互点采集事件，统一携带 `page_id`、`referrer`、`component_id`、`position` 等维度，形成可复用的页面/组件字典。
- 指标口径：PV/UV/CTR 的计算口径由数仓侧沉淀，本侧对齐事件字段与维度，并在查询层提供常用筛选（时间范围、空间、渠道、页面、组件）。
- 可视化交互：提供趋势图、TopN 排行、漏斗（曝光->点击）与多维筛选；支持从聚合图表下钻至“页面访问明细/点击明细”。

#### 会话维度分析

##### 聚类构建逻辑（基于语义相似图的聚类）【重要】
目标：对历史 Query 进行语义聚合，形成“主题簇（Cluster）”，用于会话层面的结构化分析。
技术实现思路：
1. 对历史 Query 进行向量化编码，写入向量检索索引（RAG 向量库）
2. 设定 `min_score`，相似度 ≥ 阈值的 Query 之间建立语义连接
3. 构建相似图，每个 Query 为节点，相似度满足阈值则建立边
4. 形成聚类，从未归类 Query 出发，递归召回满足阈值的 Query，直到无新增节点，此时形成一个聚类
5. 使用 LLM 总结主题，可清洗低相关 Query，生成聚类标题（主题标签）
6. 重复以上步骤，直到所有query都被聚类。
7. 新 Query 尝试判断是否属于已有的聚类若匹配则归入，否则创建新聚类
