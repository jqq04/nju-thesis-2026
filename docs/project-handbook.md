# 项目知识手册（写作共识版）

本文件用于记录我们对项目的共同理解：模块划分、核心数据模型、关键流程与当前实现边界。它不参与论文PDF构建，仅作为项目说明/知识手册，避免论文写作“无中生有”或偏离实际。

## 0. 背景与动机（摘要提炼）

随着互联网广告与商业化业务快速发展，围绕广告投放、产品说明、业务规范与运营实践等内容持续沉淀，形成规模庞大、类型多样的专业内容资产。这类内容广泛用于业务人员学习、运营决策支持与客户服务等场景，对业务稳定运行和效率提升具有重要支撑作用。

然而，随着业务线数量与内容规模增长，内容在管理、应用与质量保障方面逐渐暴露出问题：内容分散在不同业务线和系统中，缺乏统一的组织与复用机制；内容维护成本高；质量难以评估与治理，传统内容平台难以满足实际业务需求。

为解决上述问题，项目设计并实现了一套面向广告与商业化内容的管理、应用与治理平台。平台围绕内容全生命周期，对内容建模、组织、应用与治理进行系统性设计，为广告与商业化内容的统一管理和高效利用提供工程化解决方案。

## 2. 项目目标与范围

### 2.1 项目目标

- 将多形态内容（knowledge、faq、course、case）纳入统一内容资产底座，支持生产、发布、复用、检索与问答。
- 提供面向门户/管理后台的内容消费能力，以及面向上游AI助手的生成式问答（RAG）能力。
- 通过内容治理能力发现重复、歧义、覆盖不足等问题，并以任务化闭环推动内容持续演进。
- 通过数据洞察能力衡量内容效果与治理收益，形成闭环。

### 2.2 范围边界（当前实现）

- 内容形态：knowledge、faq、course、case 四类均纳入“内容资产”管理与发布。
- 重复度/歧义度检测：当前仅覆盖 knowledge 与 faq；course 与 case 暂不纳入模型检测范围。
- case 详情：采用 MongoDB 存储，除基础字段外其余字段统一进入 detail(JSON)。

### 2.3 核心功能点（写作提纲）

本节用于将论文摘要中的功能点沉淀为“共识清单”，后续写作可直接对照逐条展开。

#### 2.3.1 内容管理模块

- 空间与应用管理：以 Space 区分不同业务线，以 Application 表示业务线下的具体领域/使用场景，实现隔离与管理。
- 多形态内容管理：统一管理文章、问答、课程、案例，支持创建、编辑、删除、上下线与批量操作等。
- 应用引用内容：支持内容在不同应用之间被引用与复用，避免重复维护并提升一致性。
- 类目树管理：每个应用维护独立类目树，内容可挂载到类目节点用于导航与展示。
- 标签管理：为内容配置标签，实现多维度标注，支撑检索、筛选与聚合展示。
- 权限管理：区分管理后台权限与内容可见范围，满足业务隔离与合规要求。
- 审批管理：支持上线审批，审批流程与审批人可配置，保障发布过程规范化与可控。

#### 2.3.2 内容应用模块

- 页面化展示：内容列表、目录/类目、内容详情展示；关键词搜索；标签聚合展示等。
- 内容召回与生成：基于 RAG 的智能应用能力，为上游 AI 应用（如智能助手）提供高相关性的内容支撑。

#### 2.3.3 内容治理模块

- 重复度治理：检测内容相似度，识别重复问题，降低冗余。
- 歧义度治理：检测同一业务问题在不同内容间的冲突或不一致，降低歧义。
- 覆盖度治理：分析知识体系覆盖情况，发现缺失或薄弱领域。
- 线上化治理任务：将模型检测与用户反馈转化为可执行任务，指派业务人员处理并形成闭环。

#### 2.3.4 数据洞察模块

- 页面数据监控：PV、UV、点击率等指标。
- 会话维度分析：对相似 query 聚类，统计会话数量、模型识别解决率、意向转人工率与转人工率等。
- 内容维度分析：统计被召回次数、被生成次数、意向转人工次数与转人工次数等，评估内容使用效果。
- 会话明细：支持查看单会话的聚类类型、原始 query、改写 query、召回内容、是否转人工及用户相关信息等。

## 3. 术语与对象

- 空间（Space/Workspace）：业务归属与权限隔离边界。
- 应用（App/Application）：内容消费场景与配置边界（门户/智能助手/开放接口等）。
- 内容（Content）：统一内容实体，包含元数据（标题、摘要、类型、状态等）。
- 内容版本（ContentVersion）：内容的版本链，用于支持草稿/发布版本与回滚。
- 应用引用内容（App-Content Reference）：表达某应用可使用哪些已发布内容，实现跨应用复用。
- 授权策略（content_auth）：内容粒度的可见性与操作权限策略。
- 文档（Document）与切片（Slice）：索引侧对象；一个 Document 可包含多个 Slice，用于向量化与证据定位。
- RAG：检索增强生成；在问答时先召回证据片段，再基于证据生成回答并返回引用来源。

### 3.1 角色与权限术语（补充）

管理权限与内容可见性需要分别描述，以避免“后台能看到/内容能看到”混淆：

- 管理后台角色（示例）：超级管理员、空间管理员、应用管理员等，决定是否可进入后台模块与进行配置操作。
- 内容生产角色（示例）：内容编辑者、审核员等，决定是否可创建、提审、审批与发布内容。
- 内容消费角色（示例）：普通用户/客户，决定是否可在特定空间/应用下查看、检索与问答消费内容。

## 4. 模块划分（四个核心域）

从系统视角，平台由四个核心域构成：内容管理、内容应用、内容治理、数据洞察，并通过统一模型、权限体系与事件机制协同。

### 4.1 内容管理域（Content Management）

职责边界：

- 空间与应用的组织管理（创建、配置、成员与角色等）
- 内容生产与维护（创建、编辑、删除、版本管理）
- 权限控制（RBAC + content_auth）
- 审批发布与上下线
- 类目、标签等内容组织能力
- 应用引用内容能力（跨应用复用）

输出：

- “已发布内容版本”作为检索/问答/治理的唯一权威输入
- 内容变更事件，用于驱动索引更新与统计回收等异步链路

### 4.2 内容应用域（Content Application）

职责边界：

- 内容展示：目录导航、列表聚合、详情页展示
- 内容检索：关键词检索 + 筛选，结果严格受权限控制
- 生成式问答：RAG 召回与重排，返回可追溯引用来源

输出：

- 面向用户的列表/详情
- 面向上游AI助手的问答接口返回（回答 + 引用来源）

### 4.3 内容治理域（Content Governance）

职责边界：

- 质量检测：重复度/歧义度/覆盖度等
- 结果沉淀与可定位化：输出候选重复组、歧义候选、证据与得分
- 治理任务闭环：创建、分发、处理、审核、复检
- Benchmark 标注与模型迭代：通过人工标注与离线评估持续提升准确性

输出：

- 治理任务与结果
- 检测报告/指标回收数据（供洞察评估）

### 4.4 数据洞察域（Data Insights）

职责边界：

- 数据采集：页面访问、检索、问答链路日志/埋点
- 指标聚合：按空间/应用/内容/会话维度聚合
- 分析与下钻：从指标到会话明细，定位低命中、冲突口径等问题
- 治理成效统计：治理前后对比，反馈治理策略

输出：

- 指标报表与会话明细
- 治理收益评估结果，反哺治理与内容生产策略

## 5. 核心数据模型（概念级）

### 5.1 组织与资产

- Space 1..N App
- App 1..N Content（内容归属于一个源应用 source_app）
- Content 1..N ContentVersion
- App N..M Content（通过 AppContentRef 实现“应用引用内容”）

### 5.2 权限

- User N..M Role（在 Space/App 维度）
- content_auth：在“内容/应用”等资源对象上表达策略授权（mapping_type + mapping_id + tactic + variable + code）

### 5.3 索引对象

- ContentVersion 1..N Document
- Document 1..N Slice

说明：

- 索引以“发布版本”作为输入口径，保证检索/问答结果与对外展示口径一致。
- 内容与索引对象之间可通过“内容版本—文档—切片”的映射关系建立证据追溯能力。

## 6 核心流程

### 6.1 内容生命周期（生产 → 发布 → 下线）

1. 创建内容：生成 Content + 草稿 ContentVersion
2. 编辑内容：形成新的草稿版本或更新草稿版本
3. 提交审核：进入待审批状态
4. 审批通过：生成发布版本，内容状态变为已发布
5. 下线：内容对外不可见，但保留版本与审计信息

关键约束：

- 对外可用内容以“发布版本”为准
- 发布动作会触发后续异步链路（索引更新、统计回收等）

### 6.2 发布后的事件驱动处理（索引/统计）

典型事件：

- ContentPublished：内容发布版本产生
- ContentOffline：内容下线

下游处理（异步）：
- 索引构建：更新全文索引（Es）与向量索引（Embedding），产出 Document/Slice
- 数据洞察：记录发布影响范围、后续消费链路数据聚合

### 6.3 检索与问答（RAG）

查询阶段的通用链路如下（对门户检索与问答均适用，差异在于是否进入生成阶段）：

1. 确定范围：空间/应用配置 + 调用方身份（用户/AI助手）。
2. 权限过滤：基于 RBAC + content_auth 对候选范围与结果进行过滤。
3. Query 改写（可选）：LLM(flash) 对用户输入进行优化/标准化，例如补全上下文、统一术语或消歧。
4. 召回：全文检索/向量检索，多通道召回。
5. 重排：去重、融合排序，得到证据片段集合（Slice）。
6. 生成（问答场景）：LLM 基于证据片段生成回答。
7. 返回：检索返回结果列表；问答返回“回答 + 引用来源”（可追溯到内容与片段）。

#### 6.3.1 内容切片与索引构建

本节用于沉淀“不同内容形态如何入索引”的共识口径。

- 文档（knowledge）切片：以固定长度为基础切片，同时尽量保持段落语义完整，避免跨段切分导致上下文断裂。每个片段还会携带文档元数据以及所属子标题等信息。
- 问答（faq）切片：由于问答不会过长，问题与答案作为完整语义单元不进行切片处理，分别入库并建立索引，召回阶段按场景组合使用。
- 表格处理：切片时避免纯文本默认切分，按行语义完整切分，并在切片中携带对应表头信息保持结构语义。
- 图片处理：可选两种策略：1）多模态 embedding；2）使用大模型对图片生成文本化理解结果，与图片链接及上下文一并入库以提升可检索性。

### 6.4 治理：重复度/歧义度模型

当前覆盖范围：

- 参与内容类型：knowledge、faq
- 不参与：course、case（暂不支持）

术语口径：

- 内容相似：两篇内容描述的主题相似或存在包含关系。
- 歧义包含两类：单文本语义歧义（多义/指代不明等）与跨文本描述冲突歧义（同一实体描述不一致）。

工作流程（避免全量两两比较）：

1. 遍历待检测内容
2. 候选召回：对每个内容在检索侧召回 topk 相似内容（向量召回）
3. 大模型判别：对“当前内容—候选内容”构造 prompt，调用 LLM 判断相似度/歧义度并输出结果（含得分/解释）
4. 结果汇总：形成相似对、重复组、歧义候选组等结构化结果
5. 产出：输出检测结果，用于人工复核与治理任务创建

判定与运营口径：

- 重复判断：文章/问答相似度大于等于 50% 判为重复，并导出数据推动治理。
- 重复等级：完全重复、包含重复、极度重复（80%–100%）、中度重复（50%–80%）、轻微重复（25%–50%）、不重复（<25%）。

输出建议（概念）：

- content_id_a, content_id_b
- score_sim / score_amb
- reason（LLM 输出的可解释文本）
- evidence（关键片段引用，用于人工复核）

### 6.5 治理：覆盖度机制

覆盖度用于刻画“在给定阈值下，用户原声是否能够召回到相关知识”，用于持续发现“缺知识”与“薄弱领域”，并将其转化为可执行、可闭环的补充治理。

#### 6.5.1 覆盖度机制（基于用户原声）

该机制以真实用户会话为驱动，面向“当前业务真正遇到的问题”发现覆盖缺口。

1. 阈值配置。管理员配置知识召回阈值（阈值越高，召回越精准）。
2. 缺知识识别。在指定会话周期内，对用户原声（query）执行召回评估：当召回结果未满足阈值要求（例如无召回结果或召回得分不足）时，视为“缺知识”；满足阈值要求则视为“有知识”。
3. 聚类与任务生成。对缺知识原声进行聚类，相似原声聚为一类，生成覆盖任务（任务包含缺口主题、代表性原声样本与影响范围等信息）。
4. 任务处理与回收。业务人员对覆盖任务进行处理（补充内容/完善现有内容/确认无需补充等），并记录处理结果。
5. 每日巡检与自动闭环。系统每日定时巡检覆盖任务对应的缺知识样本，若在当前阈值下已可召回到相关答案，则自动将任务标记为已完成，避免“内容已补充但任务未更新”导致统计失真。

#### 6.5.2 内容先验覆盖模型（基于知识图谱 Diff）

除用户原声驱动的覆盖治理外，平台同时探索基于“业务知识基座”与“内容中台知识库”的先验对比能力，用于更主动地发现缺失、过时与潜在错误内容。

核心思路：

- 对两套知识源分别构建知识图谱，并通过图谱 Diff 发现内容中台相对基座的缺失实体/关系、过时实体/关系等差异，形成可治理的问题清单。

整体流程（概念）：

1. 获取数据源：从业务知识基座与内容中台抽取文档与元信息，并进行必要的状态标记与清洗。
2. 图谱构建：分别为两侧知识源创建数据集并触发图谱构建，支持全量构建与按更新时间驱动的增量构建。
3. 图谱数据沉淀：将图谱结果以文件形式落地（例如存储到对象存储），并将图谱元信息落库，便于后续查询与 Diff 调度。
4. 图谱 Diff：
   - 将实体与关系三元组转化为文本表示；
   - 生成 embedding 并计算语义相似性，完成跨图谱实体匹配；
   - 输出差异实体与差异关系，形成“缺失/过时”候选集合。
5. 治理转化：将 Diff 的差异结果转化为覆盖任务或其他治理任务，进入线上任务闭环，并通过复检与指标回收评估治理收益。

### 6.6 质量治理任务线上化

平台将质量治理从“离线计算+人工推动”升级为“线上任务化闭环”：将治理过程抽象为主任务与多类子任务，并通过人工处理与系统巡检保证闭环可追踪、可统计。

#### 6.6.1 主任务（上层管理单元）

- 职责：负责治理任务的创建、配置与状态管理，并承载一次治理计算的上下文。
- 创建流程：用户在空间层新增“任务”菜单，选择“新建任务”，填写任务名称、选择业务空间、设置会话周期与召回阈值。
- 状态流转：系统执行内容补充任务计算后，主任务状态变为“计算中”；计算完成后变为“计算完成”，并生成治理任务列表。

#### 6.6.2 覆盖任务

- 处理机制：
  - 人工处理：在任务列表页进入“任务处理”，选择“无需补充/需要补充”，并填写说明。
  - 系统巡检：每日定时巡检，对“待处理”任务检查系统内是否已有知识；若已存在则自动标记为“已完成”。

#### 6.6.3 重复歧义任务

- 任务目标：识别并处理内容重复、歧义问题，判定规则沿用既有跑数规则。
- 生成口径：对比“本应用下全部上线状态的文章、问答”与“内容中台全量上线状态的内容”，生成重复内容对与歧义内容对。
- 处理机制：
  - 人工处理：人工判断是否需要内容下线，完成任务标记；若需要下线，可下线本应用下重复的内容。
  - 系统巡检：每日定时巡检，检查自上次巡检后发生内容变更/下线/删除的内容是否关联待处理任务；若不再存在重复/歧义，则自动标记为“已完成”。

### 6.7 内容质量体系

内容质量体系围绕“内容覆盖、内容质量、效果评估”三大方向，以“客观质量+主观反馈”为度量标准，结合数据驱动与人机协同，构建可量化、可监控、可迭代的质量闭环。

#### 6.7.1 内容覆盖机制

- 痛点：覆盖维护依赖人工后验检查，验证成本高；覆盖率依赖 CQC 人力标注与季度补充，难以覆盖全量知识。
- 方案：通过召回阈值判定“有/缺知识”，对缺知识原声聚类生成覆盖任务，并通过每日巡检保障任务状态与完成率统计。

#### 6.7.2 内容去重机制

- 模型侧：建设内容重复识别模型与 benchmark 数据集，通过多轮标注与模型优化，识别准确率达到 90%+。
- 运营侧：模型判断文章/问答是否重复，相似度大于等于 50% 判为重复，导出数据推动各业务下线重复内容。

#### 6.7.3 内容一致性机制（歧义）

- 模型侧：建设内容歧义识别模型与 benchmark 数据集，识别准确率要求达到 80%+。
- 运营侧：模型判断文章/问答间是否存在歧义，流程与去重机制一致。

#### 6.7.4 数据分析提效

- 支持查看原声分布：对全量用户原声进行聚类分析。
- 支持会话解决效果与内容应用效果的分析洞察，辅助内容优化与治理策略调整。

### 6.8 Benchmark 标注与模型迭代


1. 样本抽取：从历史内容库与既有治理任务中抽样
2. 人工标注：标注“重复/不重复”“歧义/不歧义”以及score等标签，形成 benchmark
3. 离线评估：对召回策略、topk、prompt、模型版本、模型参数进行对比评估
4. 灰度上线：将更优组合配置化上线，逐步扩容覆盖范围
5. 持续校准：结合复检结果与反馈数据调整判别口径
